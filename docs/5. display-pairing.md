# Display Pairing Flow

1. Display boots
2. Creates a `displayPairingRequest` record. Receives a request Id and a human-readable pairing code.
3. Received a request Id and start listening on channel `pr-[request-id]`
4. Admin chooses a display and enters the pairing code.
   1. The server looks for a pairing request record, using the pairing code, ensures the status is `pending` and that the record has not expired.
   3. The display record status is set to `paired`
   4. The request Id is stored on the display record
   5. The pairing record status is set to `paired`
   6. The server broadcast a `paired` event back on the channel, passing the display Id.
5. The display received the notification and calls the `POST: /displays/{display-id}/session` endpoint, passing the request Id in the body.
6. The server validates the pairing request and creates a random session Id (uuid) and stores it on the display record, then returns it to the display client.
   1. Validate that the `requestId` in the body matches the the one stored on that display record
   2. The pairing record status us `paired`
7. The display status is changed to `connected`, link to pairing record is removed
8. The pairing record is deleted
9. The display passes the session token as bearer token in consecutive requests
